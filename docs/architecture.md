# Архитектура «Мужская опора»

Статус: **согласуемый план до начала разработки**
Дата: 2026-07-16

## 1. Цель и границы версии 1.0

Продукт — безопасное сообщество для совершеннолетних мужчин: профиль без паспортных данных, поиск собеседников, личные и групповые чаты, аудио/видеовстречи, жалобы и модерация.

Версия 1.0 включает:

- регистрация по e-mail или телефону с подтверждением, восстановление доступа;
- профиль: отображаемое имя, возрастной диапазон/дата рождения, город по желанию, языки, описание, интересы и настройки видимости;
- рекомендации анкет и начало личного диалога;
- личные чаты и группы до 10 участников;
- аудио- и видеокомнаты до 10 участников по приглашению;
- жалобы, блокировки, модерация и аудит действий;
- адаптивный web-интерфейс (desktop, планшет, мобильный браузер).

Не включать в версию 1.0 без отдельного решения: запись звонков, поиск по содержимому личных сообщений, публикацию психологических диагнозов, нативные iOS/Android-приложения и платные подписки. Это не «урезание демо», а осознанные отдельные продукты с собственными правовыми, финансовыми и эксплуатационными требованиями.

## 2. Главное решение

Начать с **модульного монолита**, а не с набора микросервисов:

```text
Браузер / PWA
      │ HTTPS, WebSocket, WebRTC
      ▼
Next.js (web) ──────────────► NestJS API + WebSocket Gateway
      │                              │        │
      │                              │        ├── Redis (pub/sub, presence, rate limits)
      │                              ├────────── PostgreSQL (источник истины)
      │                              ├────────── BullMQ workers (фоновые задачи)
      │                              └────────── S3 storage (аватары/вложения)
      │
      └──► LiveKit SFU ──► coturn / публичная сеть
                    ▲
          короткий JWT выдаёт только NestJS
```

Это полноценная production-архитектура без преждевременных микросервисов. При росте независимо масштабируются web, NestJS/WebSocket, фоновые воркеры и LiveKit. Внутренние модули нельзя связывать прямыми запросами к таблицам друг друга: только через сервисный слой и доменные события.

## 3. Стек

| Зона | Выбор | Зачем |
| --- | --- | --- |
| Клиент | Next.js, React, TypeScript, TanStack Query, Tailwind CSS + локальные UI-компоненты | типобезопасный адаптивный интерфейс, SSR для публичных страниц, удобная PWA-эволюция |
| API | NestJS (Fastify adapter), TypeScript, OpenAPI | единый язык с frontend, модульная бизнес-логика, validation и typed API |
| Административный UI | отдельный защищённый раздел Next.js | модерация — продуктовый интерфейс с нужными сценариями, а не доступ к сырым таблицам |
| Realtime-чат | NestJS WebSocket Gateway (Socket.IO) + Redis adapter | двусторонние события, reconnect, комнаты и несколько экземпляров API |
| Фоновые задачи | BullMQ + Redis | уведомления, обработка изображений, пересчёт рекомендаций, антиспам |
| База | PostgreSQL | транзакции, полнотекстовый поиск, индексы, JSONB там, где это оправдано |
| Медиа | self-hosted LiveKit + coturn | SFU для аудио/видео до 10 участников; не нужно писать WebRTC-инфраструктуру с нуля |
| Файлы | S3-совместимое хранилище (MinIO локально, объектное storage в production) | приложение не хранит пользовательские файлы на своём диске |
| Наблюдаемость | Sentry, Prometheus/Grafana, Loki/структурированные логи | ошибки, нагрузка и расследование инцидентов |
| Деплой | Docker Compose для разработки и малого production; затем Kubernetes/managed-сервисы по фактической нагрузке | одинаковая среда разработки, без преждевременной сложности |

**Почему не Django:** он не нужен для LiveKit, чата или PostgreSQL. Его стандартная админка полезна, но модерации всё равно потребуется свой UI: очередь жалоб, контекст переписки, санкции и аудит. TypeScript на frontend и backend уменьшает число моделей, SDK и специалистов. Django остаётся разумной альтернативой только если стандартная back-office-админка важнее единого стека.

## 5. Домены и ключевые сущности

| Модуль | Сущности | Ответственность |
| --- | --- | --- |
| accounts | User, Identity, Session, Role, Consent | вход, подтверждение, сессии, согласия |
| profiles | Profile, Language, Interest, ProfileVisibility | публичная анкета и приватность |
| matching | Recommendation, MatchFeedback, Block | выдача кандидатов и исключения |
| messaging | Conversation, Membership, Message, Attachment, ReadCursor | постоянное хранение чата и права доступа |
| calls | CallRoom, CallParticipant, Invite, CallEvent | бизнес-правила комнаты; LiveKit не источник истины |
| moderation | Report, ModerationCase, Sanction, AuditEvent | жалобы, решения, бан, неизменяемый аудит |
| notifications | Notification, DeliveryAttempt | уведомления и повторная доставка |
| billing | Product, Entitlement, Payment, Subscription | будущие услуги, изначально отключённый модуль |

`User` — собственная сущность приложения с первого дня. В ней минимум данных. Отображаемое имя хранится отдельно от метода входа; e-mail/телефон никогда не выдаётся другим пользователям.

## 6. Основные сценарии

### Рекомендация и первый диалог

1. Пользователь завершает минимальный профиль и выбирает интересы.
2. `matching` отбирает только доступные анкеты: 18+, совместимый язык, не заблокированные, не скрытые, без активной санкции.
3. Ранжирование сначала прозрачное и простое: общие темы, язык, приемлемая возрастная дистанция, свежесть активности. Оно не делает психологических выводов.
4. «Написать» создаёт или возвращает единственный direct-conversation для пары пользователей. Повторный клик идемпотентен.
5. Любой пользователь может скрыть рекомендацию, заблокировать собеседника или пожаловаться.

### Сообщение

1. Клиент отправляет HTTPS-команду с `client_message_id`.
2. NestJS проверяет участника, лимиты, блокировки и сохраняет сообщение в PostgreSQL транзакционно.
3. В ту же транзакцию записывается outbox-событие; после commit оно публикуется через Redis в WebSocket-группу.
4. Получатели получают событие `message.created`; при переподключении забирают историю cursor-пагинацией из API.

Это исключает потерю сообщений при падении WebSocket и дубли при повторной отправке.

### Аудио/видеовстреча

1. Создатель выбирает существующий групповой чат или создаёт разовую комнату.
2. `calls` фиксирует `CallRoom` с лимитом 10, правилами доступа и временем жизни.
3. Участник запрашивает `POST /calls/{id}/join`; NestJS проверяет права, бан и лимит, затем выдаёт короткоживущий JWT LiveKit с identity вида `user_<uuid>` и ровно нужными grants.
4. Браузер подключается напрямую к `wss://rtc.example.ru`, публикует/подписывает треки через LiveKit.
5. Подписанный webhook LiveKit обновляет технический статус комнаты, но не заменяет бизнес-проверки NestJS.

Токены нельзя выпускать в Next.js и нельзя делать бессрочными. Для закрытия встречи NestJS отзывает доступ через LiveKit server API и запрещает выпуск новых токенов.

## 7. Realtime и LiveKit

LiveKit — подходящий выбор: это self-hosted SFU для многопользовательского WebRTC с JWT, SDK, TURN/TCP/UDP и горизонтальным режимом. Его репозиторий и документация: <https://github.com/livekit/livekit>.

Правила эксплуатации:

- media-трафик не проходит через NestJS/Next.js;
- LiveKit, PostgreSQL и Redis — разные контейнеры/сервисы, с отдельными секретами;
- DNS: `app`, `api`, `rtc` на одном домене; TLS обязателен везде;
- открыть для LiveKit только документированные TCP/UDP-порты, TURN — на отдельном hostname; firewall по принципу deny-by-default;
- отключить запись звонков в версии 1.0; для записи нужны явное согласие всех участников, retention и отдельный бюджет;
- проверять работоспособность через LiveKit CLI/load test до релиза, включая сеть с мобильного интернета.

Один VPS 4 vCPU/8 GB годится лишь для раннего пилота с малым числом одновременных встреч. Его нельзя обещать как гарантию «10 человек в комнате × много комнат»: SFU отправляет видео каждому подписчику, поэтому лимит определяется реальным исходящим трафиком, кодеками, simulcast и качеством сети. Перед публичным запуском нужна нагрузочная проверка, а при росте — отдельный узел LiveKit и managed PostgreSQL/Redis.

## 8. Безопасность, модерация и персональные данные

- Вводить только необходимые данные; дата рождения и контакты скрыты от других пользователей.
- Публичное имя, город, описание и интересы имеют отдельные настройки видимости.
- Хеширование паролей Argon2id, MFA для администраторов, httpOnly/secure cookies, CSRF, CORS allow-list, ограничение частоты входа и сообщений.
- Все действия модераторов и санкции — в append-only `AuditEvent`; удаление аккаунта обезличивает профиль, но не ломает целостность чатов.
- Жалоба из сообщения сохраняет техническую ссылку на конкретное сообщение; доступ к нему получают только роли модерации.
- Автофильтры дают только приоритет очереди: блокировать пользователя исключительно по модели нельзя. Для угрозы жизни — показать кризисный сценарий и контакт экстренных служб соответствующего региона.
- До сбора данных согласовать политику конфиденциальности, правила сообщества, возрастное ограничение, сроки хранения и договоры с провайдерами. Нужна проверка юристом по юрисдикции запуска, особенно если проект позиционируется как психологическая помощь.

## 9. Масштабирование без переписывания

| Этап | Развёртывание | Триггер перехода |
| --- | --- | --- |
| Стартовая production-среда | reverse proxy, web, NestJS, worker, Redis, PostgreSQL; LiveKit отдельно логически | до подтверждённой нагрузки и первых пользователей |
| Рост | web и API в нескольких экземплярах; Redis/PostgreSQL вынести в managed/отдельные узлы; отдельный LiveKit + coturn | рост latency, очередей, CPU/egress или отказоустойчивость |
| Масштаб | контейнерный оркестратор, несколько LiveKit nodes, балансировка и региональная стратегия | реальные метрики подтверждают необходимость |

Не масштабировать «на глаз». SLO версии 1.0: API p95 < 400 ms, доставка realtime-события p95 < 1 s, error rate < 1%; для звонков — успешное подключение, packet loss и bitrate по типу сети.

## 10. Что подготовлено для будущих платных услуг

Платёжный код не нужен в версии 1.0, но контур заранее отделён:

- доступ определяется `Entitlement`, а не условием «пользователь оплатил» по всему коду;
- функции проверяют policy (`can_create_extended_room`, `can_access_expert_session`), а не тариф напрямую;
- платежи обрабатываются webhook-ами провайдера идемпотентно; сумма и статус никогда не принимаются из браузера;
- бухгалтерские и юридические требования уточняются до подключения конкретного провайдера.

Это позволит добавить подписку, донат или платные встречи без миграции логики чатов и звонков.

## 11. Нефункциональные требования

- Mobile-first: контрольные ширины 360, 768, 1024, 1440 px; управление звонком доступно большим пальцем; клавиатурная навигация и контраст WCAG AA.
- Контент без бесконечной загрузки: cursor pagination, виртуализация длинных списков чата.
- `id` — UUID/ULID, время — UTC; soft-delete там, где нужны аудит и восстановление.
- Миграции БД, линтеры, type-check, unit/integration/E2E-тесты — обязательны в CI.
- Бэкапы PostgreSQL ежедневно, проверка восстановления ежемесячно; S3 versioning/lifecycle для вложений.
- Секреты только в secret store/переменных окружения, никогда в Git и фронтенде.

## 12. Решения, которые нужно утвердить до кодинга

1. Юрисдикция, возрастное ограничение и формулировка сервиса: сообщество поддержки или лицензированная психологическая услуга.
2. Метод регистрации: e-mail, телефон или оба; нужен ли анонимный ник вместо реального имени.
3. Открыты ли группы для поиска или доступны только по приглашению.
4. Нужны ли вложения в чате в первом релизе — они заметно расширяют антивирусную и moderation-задачу.
5. Регион серверов и бюджет пилота; отдельно бюджет на исходящий трафик LiveKit.
6. Нужны ли push-уведомления в версии 1.0 и какие каналы допустимы.
